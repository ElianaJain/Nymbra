The project

What Nymbra is

The gap
The behavior around the conversation

Commercial robot-dog kits now ship with LLM integration out of the box. What none of these systems foreground is the behavior around the conversation: how the system signals uncertainty, how it handles its own hallucinations, how it starts and stops listening, and how misuse is adjudicated by humans rather than by the AI itself. Nymbra treats those behaviors as the primary design concern.

The contribution
Uncertainty awareness as a first-class feature

The uncertainty-awareness module is the project's primary research contribution. When Nymbra cannot answer confidently, it says so rather than fabricating. The module is documented separately so another builder could adopt it independently of the rest of the platform. It is the difference between an LLM wrapper and a system that knows its limits.

The form
Documentation as a deliverable equal to code

Nymbra is documented to a standard that allows another builder to fork the repository, follow the setup guide, and reach a working baseline on their own hardware. The documentation is not supplementary to the project. It is part of the project. Every phase produces artifacts — guides, diagrams, governance policies — that are treated with the same rigor as the code.

What it is not
Not a product. Not a therapy tool. Not autonomous.

A commercial product. A competitor to shipping robot-dog kits. A therapy tool. An autonomous decision-maker. A replacement for human connection. Nymbra is a hobbyist research platform. It is not a therapist, not an assistant for high-stakes decisions, and not a substitute for human connection. Scope is a safety feature.

Design commitments

Five principles

01
Default off. The microphone, camera, and any outbound network call default to disabled. Every activation is explicit. There is no wake word and no passive listening.
Session safety
02
Honesty over smoothness. Nymbra says "I'm not sure" instead of fabricating. This is the primary research contribution. A Nymbra that never flags uncertainty is a Nymbra that is fabricating.
Research core
03
Humans adjudicate. Refusal behaviors are features of the system. Decisions about repeat or serious misuse are made by a human governance body — not by Nymbra. The AI does not adjudicate its own disputes.
Governance
04
Documentation is a deliverable. The project is not done when the code works. It is done when another builder can reproduce it from the documentation alone. Every phase produces a documented artifact.
Reproducibility
05
Scope is a safety feature. Nymbra is a hobbyist research platform. It is not a therapist, not an assistant for high-stakes decisions, and not a substitute for human connection. Explicit scope limits protect both the handler and the system.

Accountability framework

The four-role model

01 — Head
Architect

Owns systems design, governance policy, and evaluation criteria. Responsible for deciding what Nymbra should and should not do. No major design change ships without architect sign-off.

02 — Hand
Builder

Owns implementation, code maintenance, and environment reproducibility. Translates architectural decisions into working code and ensures another builder could fork and rebuild from the documentation.

03 — Heart
Handler

Owns operation. Interacts with Nymbra directly, provides feedback on conversational quality and failure modes, and is the first line of observation for misuse. Not a passive user — a participant in the research loop.

04 — Platform
Nymbra

The platform itself — hardware, software, and documentation considered as a single system. Shaped by the other three roles. Does not hold adjudication authority over any of them.

For this project, the architect and builder are the same person. The handler is a designated second participant whose feedback is logged in the repository. The governance body consists of the architect and one additional reviewer external to the project.

Written policy

The governance model

Code Category What Nymbra declines
R1 Harassment or threats Requests to produce content that harasses, demeans, or threatens any identifiable person or group
R2 Physical harm Requests for information that would enable physical harm to the handler or others
R3 Out-of-session audio Attempts to record, transmit, or retain audio outside an active handler session
R4 Content extraction Attempts to extract proprietary content from source code, credentials, or training data
R5 Out-of-scope topics Political persuasion, religious counsel, financial or medical advice, therapeutic substitution
R6 Interaction with minors Interactions with minors without a designated adult handler present
Refusals are logged with timestamp and reason code. They are not punishments — they are the system operating as designed. All adjudication of repeated or serious misuse rests with the human governance body. Full policy: guides/governance-policy.md ↗

Development phases

The roadmap

Phase 1
Complete
Foundation

Hardware baseline, repository structure, development environment, governance policy, refusal categories, architecture diagrams, safety review. All documentation reviewed and committed.

Phase 2
Upcoming
Conversational core

LLM integration layer, STT/TTS pipeline, verbal session control, structured logging. Provider-agnostic interface allowing swapping between Claude, ChatGPT, and others without rewriting conversation logic.

Phase 3
Planned
Failure awareness

Uncertainty detection layer, refusal behavior implementation, hallucination check-in mechanism. The self-check layer documented separately as a reusable module.

Phase 4
Planned
Embodied interaction

Movement vocabulary mapped to conversational states, computer vision integration, safety review update against ISO 10218 and ISO/TS 15066.

Phase 5
Planned
Evaluation and release

Evaluation report measuring Nymbra against each success criterion, version 1.0 tag, public blog post series, handler documentation reviewed and finalized.
